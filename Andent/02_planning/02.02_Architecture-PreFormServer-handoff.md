## 1. Overview

This document focuses specifically on the **handoff process** from Andent Web to PreFormServer - the critical boundary where prepared STL jobs transition from the web application's queue to the Formlabs print infrastructure.

### Key Principle

**Andent Web prepares. PreFormServer executes. Formlabs Dashboard tracks.**

Andent Web's responsibility:
1. **Intake & Classification** - Upload, classify, review cases
2. **Build Planning** - Group whole cases into Form 4B/Form 4BL manifests by compatible printer group/material/layer-height
3. **Handoff** - Send build manifests to PreFormServer with per-file preset hints
4. **Status Display** - Poll Formlabs Web API to display job status

PreFormServer handles: orient, pack, supports, dispatch to printers.
Formlabs Dashboard/Web API handles: cross-printer job tracking, screenshots, status updates.

### Current Repository Snapshot (2026-04-27)

- Implemented in code: compatibility-aware Form 4B/Form 4BL build manifests, job naming, scene creation/import/send calls, preset-hint propagation, explicit scene settings, auto-layout validation, whole-case rollback, row and bulk printer-group routing, density-based held builds, `print_jobs` persistence, Print Queue endpoints/UI, Formlabs status sync, and screenshot caching.
- Verified in repository: `/api/uploads/classify`, `/api/uploads/rows/send-to-print`, `/api/uploads/rows/bulk-update`, planning preview, print-queue polling, screenshot caching, Form 4B/Form 4BL planner tests, held-build release path tests, the full pytest suite (`250 passed` with plugin autoload disabled in this environment), TypeScript checking, and the affected Playwright bulk-actions spec (`2 passed`).
- Still not yet externally validated here: a live PreFormServer + Formlabs cloud round-trip against real services and printers.

### Planner Calibration Snapshot (2026-04-24)

Dedicated packing-algorithm reference: `Andent/02_planning/02.03_Architecture-Packing-Algorithm.md`.

- Fit decisions remain heuristic.
- Planner fit math now uses raw `x_mm * y_mm` for non-full-arch models.
- Geometry-detected Form 4BL full-arch models use a calibrated reduced effective footprint.
- Planner fit math no longer uses tooth support inflation.
- The Form 4BL evaluation benchmark now uses the live XY platform area (`353 mm x 196 mm = 69,188 mm^2`) as the Form 4BL planner budget. It improved from 11 planned builds to 4 planned builds on `From 4BL Test Data`, with model counts `[29, 34, 35, 24]` close to the Human Pack benchmark `[31, 36, 36, 22]`.
- The historical calibrated 4-build export rerun saved all 4 local Form 4BL `.form` files and screenshots in `Andent/02_planning/98_VerificationArtifacts/full_arch_calibration_20260424/form4bl_form_exports_calibrated_4build/`. It used `model_spacing_mm = 0` and `allow_overlapping_supports = false`; build 2 failed with default/implicit spacing and succeeded with zero spacing. The target production contract now uses explicit `model_spacing_mm = 1`, so dense builds need fresh evidence after implementation. The `.form` files are generated binaries and are intentionally gitignored; screenshots and JSON summaries are tracked.
- The calibrated generated export averaged `48.46%` mesh-projection XY density versus `56.13%` for Human Pack and `22.01%` for the earlier generated 9-build set.
- Print validation remains a publish-readiness risk: all 4 generated exports still report support/cup validation issues, so these artifacts prove packing/export behavior rather than final print-ready support validation.

```
│                    PHASE 1: PREFORMSERVER HANDOFF FLOW                 │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  STEP 1: BATCH CASES BY PRESET                                          │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  Group Ready cases by preset:                                   │   │
│  │  • Ortho Solid → Batch_001                                      │   │
│  │  • Tooth → Batch_002 (auto-supports)                            │   │
│  │  • Die → Batch_003                                              │   │
│  │  Rule: One case cannot span multiple batches                    │   │

│                              ↓                                          │
│  STEP 2: CREATE PRINT JOBS                                              │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  For each batch:                                                │   │
│  │  • Generate job name: YYMMDD-001, YYMMDD-002...                  │   │
│  │  • Create scene in PreFormServer                                │   │
│  │  • Import all case STLs into scene                              │   │
│  │  • Configure preset (supports, orientation)                     │   │
│  │  • Send to printer group                                        │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                              ↓                                          │
│  STEP 3: TRACK IN PRINT QUEUE                                           │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  Andent Web Backend → Formlabs Web API                          │   │
│  │  • Poll every 5s for status updates                             │   │
│  │  • Cache: 5 second TTL                                          │   │
│  │  • Display: Screenshot, Job Name, Cases, Status                 │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                              ↓                                          │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  PreFormServer → Formlabs Printers                              │   │
│  │  • Orient & Pack                                                  │   │
│  │  • Generate supports (Tooth only)                                 │   │
│  │  • Print production                                               │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.2 Holding For More Cases

Phase 1 includes a queue-efficiency hold state before dispatch.

Operator intent still starts in the Active queue: the operator selects rows and clicks **Send to Print**. Those rows leave Active and enter the existing In Progress flow. If the final planned build for a compatibility group is below the global density target, Andent holds that build instead of dispatching immediately while the office cutoff has not passed.

Hold/release contract:

| Rule | Contract |
|------|----------|
| Density metric | Planner estimated density: `sum(effective_xy_footprint) / printer_xy_budget` |
| Density target | Single global setting, default `40%` |
| Cutoff | Single global local-office setting, default `18:00` |
| Hold scope | Only the final below-target build per compatibility group |
| Compatible intake | Only rows the operator later sends to print may merge into a held build |
| Replanning | Held rows plus newly sent compatible rows are replanned together, while respecting planner budgets |
| Release before cutoff | Dispatch when density reaches target, or when the operator uses **Release now** |
| Release at cutoff | Dispatch held builds even when still below target |
| After cutoff | New below-target builds dispatch immediately |
| Startup after cutoff | Persisted held builds wait for operator action; no surprise dispatch on restart |

Held builds are visible as **Holding for More Cases** while their rows are In Progress. The UI shows compatibility group, current estimated density, target density, cutoff time, included cases, hold/release metadata, validation warnings when present, and a **Release now** action.

Persisted hold metadata includes the held build ID, compatibility key, printer group, material label, layer height, row IDs, case IDs, manifest JSON, estimated density, density target, cutoff time/date, hold/release reason, operator-release flag, validation result, validation errors, and timestamps.

## 3. PreFormServer API Contract

### Base Configuration

| Parameter | Value | Description |
|-----------|-------|-------------|
| Base URL | `http://localhost:44388` | Local PreFormServer instance |
| Protocol | HTTP/REST | JSON payloads, form-data for files |
| Authentication | None (local) | Assumes local trusted network |

### API Endpoints

#### 3.1 Create Scene

Creates a new dental scene for a patient case.

```http
POST /scene/
Content-Type: application/json

{
  "patient_id": "CASE12345",
  "case_name": "20260420_Case12345_Upper"
}
```

**Response:**
```json
{
  "scene_id": "scene_abc123",
  "status": "created",
  "patient_id": "CASE12345",
  "case_name": "20260420_Case12345_Upper"
}
```

#### 3.2 Import Model

Imports an STL file into an existing scene.

```http
POST /scene/{scene_id}/import-model
Content-Type: multipart/form-data

model: <binary STL file>
preset: "Ortho - Solid"  // Desired contract: optional hint for material settings
```

**Response:**
```json
{
  "model_id": "model_xyz789",
  "scene_id": "scene_abc123",
  "status": "imported",
  "dimensions": {"x": 65.5, "y": 55.2, "z": 12.3}
}
```

#### 3.3 Send to Printer

Submits a prepared scene to the printer queue.

```http
POST /print/
Content-Type: application/json

{
  "scene_id": "scene_abc123",
  "device_id": "printer_form3_001"  // Optional: specific printer
}
```

**Response:**
```json
{
  "print_id": "print_def456",
  "scene_id": "scene_abc123",
  "status": "queued",
  "queue_position": 3,
  "estimated_start": "2026-04-20T14:30:00Z"
}
```

#### 3.4 List Devices

Retrieves available printers.

```http
GET /devices/
```

**Response:**
```json
[
  {
    "device_id": "printer_form3_001",
    "name": "Form 3 - Station 1",
    "model": "Form 3",
    "status": "idle",
    "material": "Draft Resin V2"
  },
  {
    "device_id": "printer_form4_001",
    "name": "Form 4 - Station 2",
    "model": "Form 4",
    "status": "printing",
    "current_job": "print_abc789"
  }
]
```

#### 3.5 Get Scene Status

Checks the status of a scene in the print pipeline.

```http
GET /scene/{scene_id}
```

**Response:**
```json
{
  "scene_id": "scene_abc123",
```

### Current Build Manifest Contract (2026-04-27)

Andent Web now hands off planned `BuildManifest` objects rather than preset-only batches. Each manifest contains:

- `compatibility_key` for the printer group/material-label/layer-height family.
- ordered whole-case `case_ids`.
- ordered preset import groups with per-file `preform_hint` values.
- manifest JSON persisted on the print job for audit/debugging.

Rollback is also manifest-based: if PreFormServer validation rejects a candidate scene, Andent Web retries after removing the last-added whole case. If a single seed case still fails validation, the remaining rows are marked for manual review instead of splitting the case.

## 4. Client Implementation

### PreFormClient Class (Local API)

Location: [`app/services/preform_client.py`](../../app/services/preform_client.py)

```python
class PreFormClient:
    """Client for PreFormServer API interactions."""
    
    def __init__(self, base_url: str = "http://localhost:44388")
    def create_scene(self, patient_id: str, case_name: str) -> Dict[str, Any]
    def import_model(self, scene_id: str, stl_path: str) -> Dict[str, Any]
    def send_to_printer(self, scene_id: str, device_id: str) -> Dict[str, Any]
    def list_devices(self) -> List[Dict[str, Any]]
    def get_scene_status(self, scene_id: str) -> Dict[str, Any]
```

Current repository note: the implemented handoff path creates scenes from manifest scene settings, imports models with preset hints, runs auto-layout and scene validation, and routes to the row-selected printer group when one is set; otherwise it falls back to the manifest/default `Form 4BL` printer group.

### FormlabsWebClient Class (Cloud API)

New client for Formlabs Web API:

```python
class FormlabsWebClient:

```

### Usage Pattern

```python
from app.services.preform_client import PreFormClient
from app.services.formlabs_web_client import FormlabsWebClient

async def handoff_and_track(batch_id: str, stl_paths: List[str], preset: str):
    # Step 1: Handoff to PreFormServer
    with PreFormClient() as preform:
        scene = preform.create_scene(
            patient_id=batch_id,
            case_name=f"Batch_{batch_id}"
        )
        
        for stl_path in stl_paths:
            preform.import_model(scene["scene_id"], stl_path)
        
        print_job = preform.send_to_printer(scene["scene_id"])
    
    # Step 2: Track via Formlabs Web API
    web_client = FormlabsWebClient(api_token=os.getenv("FORMLABS_API_TOKEN"))
    status = web_client.get_job_status(print_job["print_id"])
    
    return {"scene_id": scene["scene_id"], "print_id": print_job["print_id"]}
```

## 5. Data Mapping

### Andent Web → PreFormServer

| Andent Web Field | PreFormServer Field | Notes |
|------------------|---------------------|-------|
| `case_id` | `patient_id` | Extracted from filename |
| `original_filename` | `case_name` | Human-readable identifier |
| `stored_file_path` | `stl_path` | Local filesystem path |
| `model_type` | Preset hint | Translated through the UI preset mapping at the PreFormServer boundary |
| `preset` | Print settings | Stored for batching and translated to a PreFormServer preset hint on import |

### Preset Contract (Phase 1 Target)

The 7 UI presets remain the product surface. A preset describes dental preparation behavior; printer target is a separate planning/routing dimension.

Default printer target is `Form 4BL`. All 7 presets are also valid on `Form 4B` when explicitly selected. The app-level printer group labels are `Form 4B` and `Form 4BL`; PreFormServer boundary code maps those stable labels to the configured lab printer group or device target.

Compatibility is based on:

```text
printer group | material label | layer height
```

Material labels are stored and compared because they are operator-readable. Material codes are API-boundary details.

| UI preset | Stable PreForm hint | Material label | Material code | Layer | Support/orientation contract |
|-----------|---------------------|----------------|---------------|-------|------------------------------|
| `Ortho Solid - Flat, No Supports` | `ortho_solid_v1` | `Precision Model V1` | `FLPMBE01` | 100 um | Imported flat orientation; auto-layout allowed; supports forbidden |
| `Ortho Hollow - Flat, No Supports` | `ortho_hollow_v1` | `Precision Model V1` | `FLPMBE01` | 100 um | STL is already hollow; no hollowing; imported flat orientation; supports forbidden |
| `Die - Flat, No Supports` | `die_v1` | `Precision Model V1` | `FLPMBE01` | 100 um | Imported flat orientation; auto-layout allowed; supports forbidden |
| `Tooth - With Supports` | `tooth_v1` | `Precision Model V1` | `FLPMBE01` | 100 um | Imported bottom-down orientation is authoritative; supports target tooth models only; support contact must stay on the bottom/support-safe side |
| `Splint - Flat, No Supports` | `splint_v1` | `LT Clear V2` | `FLDLCL02` | 100 um | Imported flat orientation; auto-layout allowed; supports forbidden |
| `Antagonist Solid - Flat, No Supports` | `antagonist_solid_v1` | `Precision Model V1` | `FLPMBE01` | 100 um | Imported flat orientation; auto-layout allowed; supports forbidden |
| `Antagonist Hollow - Flat, No Supports` | `antagonist_hollow_v1` | `Precision Model V1` | `FLPMBE01` | 100 um | STL is already hollow; no hollowing; imported flat orientation; supports forbidden |

The `*_v1` hint strings are stable API contract values. Future incompatible behavior must use a new hint such as `tooth_v2` instead of changing `tooth_v1` silently.

Scene settings must be explicit from the manifest compatibility group:

| Printer group | Material label | Layer | `machine_type` | `material_code` | `print_setting` |
|---------------|----------------|-------|----------------|-----------------|-----------------|
| `Form 4BL` | `Precision Model V1` | 100 um | `FRML-4-0` | `FLPMBE01` | `DEFAULT` |
| `Form 4B` | `Precision Model V1` | 100 um | `FORM-4-0` | `FLPMBE01` | `DEFAULT` |
| `Form 4BL` | `LT Clear V2` | 100 um | `FRML-4-0` | `FLDLCL02` | `DEFAULT` |
| `Form 4B` | `LT Clear V2` | 100 um | `FORM-4-0` | `FLDLCL02` | `DEFAULT` |

Mixing rules:

- The six `Precision Model V1` presets may share a build when printer group and layer match.
- `Tooth - With Supports` may share those builds, but support generation must target tooth models only.
- `Splint - Flat, No Supports` may share builds with other Splint cases, but cannot mix with non-Splint presets because the material differs.
- Same-case Splint plus non-Splint rows route to review in Phase 1.
- Same-case mixed printer group targets route to review.
- If bottom-only tooth support behavior cannot be verified or satisfied, the tooth case routes to review.

### Validation And Dispatch Policy

PreFormServer print validation is recorded but does not block automatic dispatch. Import, scene creation, and layout failures still block because no printable scene exists. Validation warnings should remain visible on submitted jobs.

Dispatch targets the configured printer group, not a specific physical printer. Job metadata should persist `validation_passed`, `validation_errors`, estimated density, density target, hold/release reason, and release source for audit and later calibration.

## 6. Error Handling

### Connection Errors

| Scenario | Behavior | User Feedback |
|----------|----------|---------------|
| PreFormServer unreachable | Queue rows, show "PreFormServer Offline" | Toast notification |
| Timeout (30s) | Retry once, then fail | "Connection timeout - try again" |
| Invalid response | Log error, mark row for retry | "Print service error" |

### API Errors

| HTTP Status | Meaning | Action |
|-------------|---------|--------|
| 400 Bad Request | Invalid payload | Log details, skip row |
| 404 Not Found | Scene/model not found | Check PreFormServer state |
| 409 Conflict | Scene already printing | Update local status |
| 422 Unprocessable | STL validation failed | Flag row for review |
| 500 Internal Error | PreFormServer error | Retry with backoff |

### Retry Strategy

```python
MAX_RETRIES = 3
RETRY_DELAY = 5  # seconds

for attempt in range(MAX_RETRIES):
    try:
        response = client.create_scene(...)
        break
    except ConnectionError:
        if attempt < MAX_RETRIES - 1:
            time.sleep(RETRY_DELAY * (attempt + 1))
        else:
            raise HandoffFailed("PreFormServer unavailable")
```

### Security Model

- **No authentication** between Andent Web and PreFormServer (local only)
- **API Token authentication** for Formlabs Web API (stored securely)
- **Trusted network** assumption for dental lab deployment
- **File validation** on STL import (size limits, format checks)
- **No external exposure** - Andent Web and PreFormServer bind to localhost
