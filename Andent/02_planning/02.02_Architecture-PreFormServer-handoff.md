## 1. Overview

This document describes the handoff boundary from Andent Web to PreFormServer: how prepared STL jobs leave the web queue, become PreFormServer scenes, and then appear in the Print Queue through Formlabs status sync.

### Key Principle

**Andent Web prepares. PreFormServer executes. Formlabs Dashboard/Web API tracks.**

Andent Web responsibilities:
1. **Intake and Classification** - Upload, classify, review, and override rows.
2. **Build Planning** - Plan whole-case build manifests by compatible printer, resin, and layer height, using printer-aware XY budgets and bounded case-order heuristics.
3. **Handoff** - Create scenes, import files with per-file preset hints, auto-layout, validate, and submit accepted scenes.
4. **Status Display** - Persist print jobs locally and refresh status and screenshots through Formlabs APIs.

PreFormServer responsibilities:
1. **Scene Preparation** - Build scene, import files, orient, pack, and generate supports where required.
2. **Printer Dispatch** - Submit prepared jobs to the selected printer.

Formlabs Web API responsibilities:
1. **Job Status** - Provide queue and print-state information.
2. **Screenshots** - Provide screenshot assets for the Print Queue UI.

### Current Repository Snapshot (2026-04-23)

- Implemented in code: compatibility-aware build manifests, printer-aware XY budgets, printer-specific startup case seeding (`Form 4B -> 3`, `Form 4BL -> 8`), descending-then-filler case selection, job naming, scene creation/import/send calls, preset-hint propagation, auto-layout validation, whole-case rollback, row-level printer routing, `print_jobs` persistence, Print Queue endpoints/UI, Formlabs status sync, and screenshot caching.
- Verified in repository: `/api/uploads/classify`, `/api/uploads/rows/send-to-print`, planning preview, print-queue polling, screenshot caching, planner ordering/budget tests, and the planner-related pytest surface (`55 passed` across `tests/test_preset_catalog.py`, `tests/test_build_planning.py`, `tests/test_batching.py`, and `tests/test_integration.py` with plugin autoload disabled in this environment).

## 2. Handoff Flow

```text
Ready upload rows
  -> group rows into compatibility-aware printer/resin/layer-height pools
  -> preserve whole-case grouping
  -> apply printer-aware XY budget
  -> seed new builds with the largest remaining cases
  -> Form 4B starts by attempting 3 largest cases when at least 3 remain
  -> Form 4BL starts by attempting 8 largest cases when at least 8 remain
  -> continue descending through larger cases until the first heuristic fit miss
  -> switch to smallest-case fillers for remaining budget
  -> split import groups by preset hint inside each manifest
  -> create PreFormServer scene
  -> import each STL with its mapped preset hint
  -> run auto-layout
  -> validate scene
  -> if validation fails, retry by removing the last-added whole case
  -> if a single seed case still fails, mark those rows for manual review
  -> send accepted scene to printer
  -> persist print_job row + manifest JSON
  -> frontend polls /api/print-queue/jobs every 5s
  -> backend performs cache-on-request Formlabs sync
  -> screenshot endpoint fetches and caches PNG on demand
```

### 3.5 Current Build Manifest Contract (2026-04-23)

Andent Web hands off planned `BuildManifest` objects rather than preset-only batches. Each manifest contains:

- `compatibility_key` for the printer, resin, and layer-height family.
- Ordered whole-case `case_ids`.
- Ordered preset import groups with per-file `preform_hint` values.
- Manifest JSON persisted on the print job for audit and debugging.

Planner composition now also follows a bounded, printer-aware case-order policy:

- `Form 4B` attempts the largest `3` remaining compatible cases when at least `3` are available.
- `Form 4BL` attempts the largest `8` remaining compatible cases when at least `8` are available.
- If fewer than those counts remain, the planner falls back to the existing seed-with-largest start.
- After the startup phase, the planner continues descending by case priority until the first XY-budget fit failure.
- After that first descending failure, the planner switches to smaller filler cases.
- Fit decisions remain heuristic and use the current XY-budget calculation rather than a live scene-fit probe.

Rollback is also manifest-based:

- If PreFormServer validation rejects a candidate scene, Andent Web retries after removing the last-added whole case.
- If a single seed case still fails validation, those rows are marked for manual review instead of splitting the case.

### 4.3 Usage Pattern

The current repository path is manifest-based, not preset-batch-based.

```python
from app.services.print_queue_service import process_print_manifest

def handoff_manifest(settings, manifest, manifest_rows, batch_number):
    result = process_print_manifest(
        settings=settings,
        manifest=manifest,
        rows=manifest_rows,
        batch_number=batch_number,
    )
    return {
        "scene_id": result["scene_id"],
        "print_id": result["print_job_id"],
        "job_name": result["job_name"],
        "compatibility_key": result["compatibility_key"],
    }
```

## 5. Data Mapping

### 5.1 Andent Web to PreFormServer Boundary

| Andent Web Field | Boundary Representation | Notes |
|------------------|-------------------------|-------|
| `case_id` | manifest `case_ids` and scene caller metadata | Whole-case grouping remains intact |
| `stored_file_path` | `file` | Local filesystem path is sent to PreFormServer |
| `preset` | `preform_hint` | UI preset is translated at the PreFormServer boundary |
| selected printer | `printer` | Uses row-selected printer when present; otherwise defaults to `Form 4` |
| planned build | `compatibility_key` | Persisted on `print_jobs` for audit/debugging |
| printer family | planner XY budget and startup seed count | `Form 4B` and `Form 4BL` now use different build budgets and startup-case counts |

### 5.2 Model Type to PreFormServer Preset

| Andent Model Type | PreFormServer Preset | Support Generation | Orientation | Material | Layer Height | Printer |
|-------------------|---------------------|-------------------|-------------|----------|--------------|---------|
| `Ortho - Solid` | `ortho_solid_v1` | None | Flat on platform | Precision Model Resin | 100um | Form 4BL |
| `Ortho - Hollow` | `ortho_hollow_v1` | None | Flat on platform | Precision Model Resin | 100um | Form 4BL |
| `Die` | `die_v1` | None | Flat on platform | Precision Model Resin | 100um | Form 4BL |
| `Tooth` | `tooth_v1` | Auto-generate | Auto-oriented with supports | Precision Model Resin | 100um | Form 4BL |

**Key rules**

- Only Tooth models get auto-support generation.
- All supported Phase 1 presets use Precision Model Resin, 100um layer height, and Form 4BL compatibility.
- Mixed presets may share one manifest only when they resolve to the same compatibility family.

## 6. Error Handling

### 6.1 Connection Errors

| Scenario | Behavior | User Feedback |
|----------|----------|---------------|
| PreFormServer unreachable | Handoff fails before submission | `502` surface from send-to-print endpoint |
| Timeout | Retry through client retry wrapper | User sees print-service failure if retries exhaust |
| Invalid response | Raise error and stop current manifest | Manual retry after inspection |

### 6.2 API Errors

| HTTP Status | Meaning | Action |
|-------------|---------|--------|
| `404` | Scene, printer, or resource not found | Fail current handoff |
| `409` | Conflict or busy printer | Fail current handoff |
| `422` | STL or scene validation issue | Route rows to manual review when appropriate |
| `5xx` | PreFormServer error | Retry on request failures; otherwise fail current handoff |

### 6.3 Retry Strategy

The client uses a retry decorator for request failures.

```python
MAX_RETRIES = 3
BACKOFF_FACTOR = 2.0

for attempt in range(MAX_RETRIES):
    try:
        response = client.create_scene(...)
        break
    except ConnectionError:
        if attempt < MAX_RETRIES - 1:
            time.sleep(BACKOFF_FACTOR ** attempt)  # 1s, 2s
        else:
            raise HandoffFailed("PreFormServer unavailable")
```

## 7. Status Synchronization

### 7.1 Status Mapping

| Upstream Status | Andent Web Status | Description |
|-----------------|-------------------|-------------|
| `created` | `Submitted` | Scene exists but is not yet printing |
| `queued` | `Queued` or `Submitted` | Waiting in queue |
| `printing` | `Printing` | Active print job |
| `completed` | `Completed` | Print finished successfully |
| `failed` | `Failed` | Print or validation failure |
| `cancelled` | `Failed` | Cancelled job |
| `paused` | `Paused` | Paused print job |

Once handoff occurs, detailed status tracking moves to Formlabs Web API rather than PreFormServer scene status.

## 8. Security Considerations

### 8.1 Local Deployment Model

```text
Andent Web (FastAPI)
  <-> PreFormServer (localhost by default)
  <-> SQLite + STL storage
  <-> Formlabs printers

Andent Web
  <-> Formlabs Web API (token-authenticated cloud calls)
```

### 8.2 Security Model

- No authentication between Andent Web and PreFormServer in the default local deployment.
- API token authentication for Formlabs Web API.
- Trusted-network assumption for dental lab deployment.
- File validation on STL intake before handoff.
- Default local-only binding for Andent Web and PreFormServer, with Andent Web host binding configurable for later deployment phases.
