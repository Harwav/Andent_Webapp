# Architecture: Packing Algorithm

> **Created:** 2026-04-24
> **Updated:** 2026-04-24
> **Status:** Active - planner calibration is implemented; target Form 4B/preset/holding policy is specified here and still requires implementation
> **Scope:** Phase 1 - build planning, Form 4BL packing calibration, and density evidence

---

## 1. Overview

This document is the standalone reference for how Andent Web decides which cases belong together in a print build before handoff to PreFormServer.

### Key Principle

**Andent Web plans whole-case build manifests. PreFormServer performs the physical layout.**

Andent Web does not try to solve exact 2D nesting. It estimates whether a set of cases should fit on a compatible printer by using calibrated XY footprint budgets, then sends the complete build manifest to PreFormServer for auto-layout, support validation, screenshots, and final `.form` export.

The planner is intentionally conservative about compatibility and case cohesion:

1. Keep all files for one case in the same build.
2. Mix presets only when they resolve to the same printer, resin, and layer height.
3. Estimate build fullness from XY footprint, not model count.
4. Use PreFormServer evidence to calibrate the heuristic instead of guessing from file count.

## 3. Planner Inputs And Outputs

### Inputs

The planner receives `ClassificationRow` objects after upload classification and review. The fields that matter for packing are:

| Field | Role |
|-------|------|
| `status` | Only rows with `Ready` status are planned |
| `case_id` | Used to keep each case together |
| `preset` | Resolves printer/resin/layer compatibility |
| `dimensions.x_mm` and `dimensions.y_mm` | Raw XY footprint estimate |
| `file_path` | Required for PreFormServer handoff |
| `row_id` | Used for stable import order and row-level tracking |

Rows with missing dimensions, missing row IDs, incompatible presets, or missing file paths become non-plannable manifests instead of being silently dropped.

### Outputs

The planner returns `BuildManifest` objects. A planned manifest contains:

| Field | Role |
|-------|------|
| `compatibility_key` | Printer/resin/layer grouping key |
| `case_ids` | Whole cases assigned to the build |
| `preset_names` | Presets represented in the build |
| `import_groups` | Files grouped by preset for PreFormServer import |
| `FilePrepSpec.xy_footprint_estimate` | Effective XY estimate used by the planner |
| `FilePrepSpec.preform_hint` | Preset hint sent during PreForm import |

## 4. Compatibility Model

Preset compatibility comes from `app/services/preset_catalog.py`.

The target compatibility key is built from printer group, material label, and layer height. Presets may share a build only if all three values match.

Phase 1 target groups:

| Group | Presets | Material label | Default printer | Also valid | Layer |
|-------|---------|----------------|-----------------|------------|-------|
| Model group | Ortho Solid, Ortho Hollow, Die, Tooth, Antagonist Solid, Antagonist Hollow | `Precision Model V1` | `Form 4BL` | `Form 4B` | 100 um |
| Splint group | Splint | `LT Clear V2` | `Form 4BL` | `Form 4B` | 100 um |

This means mixed model-group builds are allowed when compatible. For example, an `Ortho Solid - Flat, No Supports` model and a `Tooth - With Supports` model may share the same Form 4BL manifest, but they are still imported in preset-specific groups so PreFormServer receives the correct preparation hint for each file.

Splint rows cannot mix with non-Splint rows because they use `LT Clear V2` rather than `Precision Model V1`. Same-case Splint plus non-Splint rows route to review in Phase 1 instead of being split automatically.

Printer target is separate from preset. Same-case mixed printer group targets also route to review; whole-case cohesion takes priority over automatic splitting.

## 5. XY Fit Model

The planner uses a scalar XY budget check, not exact geometric nesting.

For each row:

```text
raw_xy_area = dimensions.x_mm * dimensions.y_mm
```

For most rows:

```text
effective_xy_area = raw_xy_area
```

For geometry-detected Form 4BL or Form 4B full-arch rows:

```text
effective_xy_area = raw_xy_area * 0.58
```

The `0.58` multiplier exists because dental arches have large empty concave regions inside their rectangular bounding boxes. Counting the full bounding rectangle caused severe under-packing, while the calibrated multiplier brought generated Form 4BL build counts close to Human Pack evidence. The same factor is the target initial behavior for Form 4B to avoid introducing a second tuning variable before Form 4B calibration evidence exists.

Tooth support inflation is no longer part of planner fit math. Tooth rows use raw bounding-box XY area. PreFormServer still owns support generation and support validation.

## 6. Full-Arch Detection

A row is treated as a full arch only when all of these are true:

```text
long_side >= 65.0 mm
short_side >= 55.0 mm
raw_xy_area >= 3400.0 mm^2
printer in {"Form 4BL", "Form 4B"}
```

This intentionally avoids preset-name-only detection. The same geometry rule can catch full arches across compatible Form 4BL and Form 4B presets, while smaller quads and tooth models stay on raw area.

Known test anchors:

- `72 mm x 68 mm` is full arch.
- `42 mm x 31 mm` is not full arch.
- `116 mm x 96 mm` on Form 4BL or Form 4B receives the `0.58` multiplier.
- Tooth rows keep raw XY area.

## 7. Case Packing Procedure

The planner works at case level.

1. Filter to rows where `status == "Ready"`, `preset` exists, and `case_id` exists.
2. Group rows by `case_id`.
3. Resolve canonical preset names and compatibility keys.
4. Reject a case as non-plannable if any row is missing required handoff data.
5. Compute each case's total effective XY footprint.
6. Sort cases by difficulty, where difficulty is based on the largest effective row plus total effective case area.
7. Group cases by compatibility key.
8. For each compatibility group, seed a build with the hardest remaining case.
9. Add startup candidates first.
10. Continue adding next-largest cases while they fit the printer XY budget.
11. When the next-largest case fails, fill remaining space with the smallest cases that still fit.
12. Emit one `BuildManifest` for the chosen cases and repeat until no cases remain.

Startup case counts are printer-specific:

| Printer | Startup case count |
|---------|-------------------:|
| Form 4BL | `8` |
| Form 4B | `3` |
| Other/unknown | `1` |

The startup pass prevents early builds from being seeded too sparsely when many similar full-arch cases are available. The later smallest-filler pass reduces wasted space without splitting cases.

## 8. PreFormServer Layout Contract

The planner's output is only the manifest. PreFormServer still performs real scene layout.

Target PreFormServer export settings are explicit from manifest compatibility. The planner should not rely on PreFormServer defaults or the first imported file.

| Setting | Value |
|---------|-------|
| Printer | `FRML-4-0` for Form 4BL, `FORM-4-0` for Form 4B |
| Material | `FLPMBE01` for `Precision Model V1`, `FLDLCL02` for `LT Clear V2` |
| Layer height | `0.100 mm` |
| Print setting | `DEFAULT` |
| Import path | `scan-to-model` |
| Units | `DETECTED` |
| Auto-layout mode | `DENTAL` |
| Models | `ALL` |
| `model_spacing_mm` | `1` |
| `allow_overlapping_supports` | `false` |

Historical calibration constraint: the calibrated 34-model Form 4BL build failed with default/implicit spacing and succeeded with `model_spacing_mm = 0`. The target production contract supersedes that with explicit `model_spacing_mm = 1` for all printers and presets. Planner budgets remain unchanged for now; future tuning should prefer adjusting the full-arch factor with benchmark evidence rather than adding another budget multiplier.

Do not treat planner success as print-readiness. A planned build still needs PreFormServer import, auto-layout, support generation where required, print validation, and operator visibility. Print validation warnings are recorded and displayed but do not block automatic dispatch under the target contract.

### Density Hold Policy

Before dispatch, Andent applies a queue-efficiency policy to the final build in each compatibility group.

| Policy | Target |
|--------|--------|
| Density metric | Planner estimated density: `sum(effective_xy_footprint) / printer_xy_budget` |
| Density target | Single global setting, default `40%` |
| Cutoff | Single global local-office cutoff, default `18:00` |
| Hold scope | Only the final below-target build per compatibility group |
| Release | Density reaches target, operator releases, or cutoff arrives |
| After cutoff | New below-target builds dispatch immediately |
| Startup after cutoff | Persisted held builds wait for operator action |

Rows in a held build have already left Active through an operator **Send to Print** action. The app must not silently pull unsent Active rows into a held build. Later sent compatible rows may be replanned with the held rows, while still respecting the same planner budget rules. If replanning produces an above-target build plus a new final below-target remainder, the above-target build dispatches and only the final remainder stays held.

## 9. XY Density Measurement

XY density is the benchmark metric used to compare generated packing against Human Pack.

The primary artifact is:

```text
Andent/02_planning/98_VerificationArtifacts/full_arch_calibration_20260424/form4bl_form_exports_calibrated_4build/xy-density-mesh-projection-025mm.json
```

Method:

1. Load the generated or human-packed `.form` scene through PreFormServer.
2. For each scene model, load the STL from PreForm's `original_file`.
3. Transform STL vertices by PreForm-reported model `position` and `orientation.z`.
4. Project transformed vertices onto XY.
5. Rasterize the projection at `0.25 mm` resolution.
6. Apply one-cell close/fill/open per model to convert surface samples into a filled footprint.
7. Union all model masks in the build.
8. Divide the union area by the full Form 4BL platform area.

Formula:

```text
mesh_projection_density_pct =
  mesh_projection_union_area_mm2 / 69188 * 100
```

Accuracy note: this raster approximation was checked against representative direct triangle rasterization. Observed difference was roughly `1-4%`, which is acceptable for packing review.

Why this metric is preferred:

- It measures the occupied projected shape rather than the full rectangular bounding box.
- It handles concave arch geometry better than summed bounding boxes.
- It can be applied consistently to generated builds and human-packed `.form` files.

## 12. Guardrails For Future Changes

Do not change any of these without rerunning the benchmark evidence:

- `FULL_ARCH_FACTOR`
- full-arch threshold constants
- Form 4BL or Form 4B XY budget
- startup case count
- smallest-filler pass
- PreFormServer `model_spacing_mm = 1` target layout setting
- density hold/release defaults (`40%` target, `18:00` cutoff)

Minimum verification for planner changes:

```powershell
$env:PYTEST_DISABLE_PLUGIN_AUTOLOAD = "1"
python -m pytest `
  tests/test_preset_catalog.py `
  tests/test_build_planning.py `
  tests/test_batching.py `
  tests/test_integration.py `
  -q
```

Minimum verification for benchmark changes:

```powershell
python -m py_compile scripts\form4bl_packing_benchmark.py
```

For publish-readiness claims, also rerun a live PreFormServer export and inspect validation errors. Passing planner tests alone is not enough.

## 13. Open Risks

| Risk | Impact | Current handling |
|------|--------|------------------|
| PreForm validation still reports support/cup issues | Saved generated forms may not be print-ready | Tracked as live validation risk |
| One scalar full-arch factor may not fit every arch shape | Some builds may under-pack or over-pack | Calibrated against Human Pack average, not exact per-shape nesting |
| Density method is raster approximate | Small measurement error | Accepted `1-4%` error range |
| `.form` files are local-only | Other machines cannot inspect exact binary scenes from Git | Regenerate through benchmark harness or copy manually |
| Form 4B uses the Form 4BL full-arch factor initially | Form 4B may under-pack or over-pack until calibrated | Use the same `0.58` factor by decision; rerun Form 4B evidence before tuning |
| `model_spacing_mm = 1` differs from the calibrated zero-spacing rerun | Dense builds may fail layout or need future factor tuning | Keep budgets unchanged for now and record layout outcomes |
| PreFormServer defaults can change | Auto-layout may regress without code changes | Record settings and send explicit scene/layout settings |
