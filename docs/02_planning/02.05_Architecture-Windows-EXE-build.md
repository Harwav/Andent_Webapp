# Windows EXE Build Guide

FormFlow ships as a single Windows executable built with PyInstaller from the tray runtime entrypoint.

## Build Surface

| Concern | Location |
|---------|----------|
| App version | `app/version.py` |
| Build script | `scripts/builders/build_windows_exe.py` |
| PyInstaller spec | `formflow.spec` |
| Runtime entrypoint | `run_formflow.py` |
| Tray/server lifecycle | `desktop/tray_runtime.py` |
| Packaged runtime resources | `app/resources/` |
| GitHub Actions workflow | `.github/workflows/build-windows-exe.yml` |
| Packaging tests | `tests/test_exe_packaging.py`, `tests/test_health_endpoints.py` |

The build script reads `app/version.py` and writes `dist/FormFlow_vX.Y.Z.exe`. Passing `--version X.Y.Z` updates `app/version.py` before the build.

`app/resources/preform-list-materials-latest.json` is bundled into the EXE and is required for local printer material codes to render as readable material names. Keep packaging tests in the build path so missing runtime data is caught before an EXE is published.

## Local Build

Run from the repository root on Windows:

```powershell
$env:PYTEST_DISABLE_PLUGIN_AUTOLOAD = "1"
python -m pytest tests/test_exe_packaging.py tests/test_health_endpoints.py -q
python scripts/builders/build_windows_exe.py --version 0.1.0
Remove-Item Env:\PYTEST_DISABLE_PLUGIN_AUTOLOAD -ErrorAction SilentlyContinue
```

Expected result:

```text
dist\FormFlow_v0.1.0.exe
```

If the build fails with `PermissionError: [WinError 5] Access is denied` for the EXE path, an existing packaged copy is probably still running. Stop only that exact EXE, then rebuild:

```powershell
$exe = Resolve-Path "dist\FormFlow_v0.1.0.exe"
Get-Process | Where-Object { $_.Path -eq $exe.Path } | Stop-Process -Force
python scripts/builders/build_windows_exe.py --version 0.1.0
```

## Packaged Smoke Test

Run the EXE on a test port without opening a browser:

```powershell
$env:FORMFLOW_WEB_PORT = "8765"
$env:FORMFLOW_WEB_OPEN_BROWSER = "0"
$exe = Resolve-Path "dist\FormFlow_v0.1.0.exe"
$p = Start-Process -FilePath $exe -PassThru -WindowStyle Hidden
try {
  $healthy = $false
  for ($i = 0; $i -lt 60; $i++) {
    try {
      $response = Invoke-RestMethod -Uri "http://127.0.0.1:8765/health" -TimeoutSec 2
      if ($response.status -eq "healthy") {
        $healthy = $true
        break
      }
    } catch {
      Start-Sleep -Seconds 1
    }
  }
  if (-not $healthy) {
    throw "EXE did not become healthy on port 8765"
  }
} finally {
  Stop-Process -Id $p.Id -Force -ErrorAction SilentlyContinue
  Get-Process | Where-Object { $_.Path -eq $exe.Path } | Stop-Process -Force -ErrorAction SilentlyContinue
  Get-NetTCPConnection -LocalPort 8765 -ErrorAction SilentlyContinue |
    ForEach-Object { Stop-Process -Id $_.OwningProcess -Force -ErrorAction SilentlyContinue }
  Remove-Item Env:\FORMFLOW_WEB_PORT -ErrorAction SilentlyContinue
  Remove-Item Env:\FORMFLOW_WEB_OPEN_BROWSER -ErrorAction SilentlyContinue
}
```

For desktop validation, launch the EXE normally and verify the tray icon appears, browser launch works, the tray menu contains the expected commands, and Quit closes the process and releases the port.

## GitHub Actions Build

The workflow at `.github/workflows/build-windows-exe.yml` already builds on `windows-latest`, smoke-tests the packaged EXE, and uploads the versioned EXE artifact.

Trigger options:

- Manual workflow dispatch with an optional `version`.
- Push a tag in `vX.Y.Z` format.
- Manual workflow dispatch with `create_release=true` to create a GitHub Release.

Release tags should match the EXE version, for example `v0.1.0` creates `FormFlow_v0.1.0.exe`.

## Release Gate

Before treating a build as release-ready, run the release gate documented in the root `README.md`. It requires a packaged EXE and a live compatible PreFormServer endpoint so the handoff path is verified against the real integration surface.
